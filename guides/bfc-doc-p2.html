bfc-doc-p2 — Phase 2 · Task 2 vertical slice

This guide expands Phase B in the enhancements playbook: one end-to-end slice of the POC Child Records Match so refresh, KPIs, and steward-facing exports are real before Task 1 polish. If you count phases numerically: A = 1 (freeze scope), B = 2 (this guide — Task 2 only), C = 3 (Task 1 slice), D = 4 (rehearsal). This is independent of document ids such as bfc-doc-j1 (Task 1 job aide).

Task 2 · Mid-review May 26, 2026
What “done” looks like for Phase 2. A scheduled (or manual) refresh completes; the model exposes POC count, per-object match rates, overall match rate, orphan count, and refresh status on one executive summary page; stewards can export an orphan table filtered to POC accounts with zero related rows across in-scope objects.

1. Entry criteria (complete Phase A first)

  • POC Master locked to logical columns SID and POC_Record_Name; template available from the hub downloads.
  • In-scope child objects confirmed in your org (API names match the catalog in enhancements.html).
  • Gateway or dataset credentials chosen for Salesforce; no secrets in M text.
  • Three critical Task 2 enhancements from the catalog selected (for example: ID normalization, SOQL batching, match scorecard).

2. Workstreams (suggested order)

A · Token and REST shell

  • Single cached OAuth or certificate path; reuse token across queries.
  • Helper function or shared query for instance URL + API version.
  • Central try / otherwise pattern so one object failure does not blank the model (see multi-object FN guide).

B · POC Id set

  • Load POC Master; trim, uppercase, validate length 15 or 18; dedupe SID.
  • Materialize POC_AccountIds as the only driver for child SOQL filters.
  • Merge POC_Record_Name back onto child results for slicers and titles.

C · Child pulls

  • SOQL IN batches (~200 Ids per request); union partitions.
  • Pagination on at least one heavy object (nextRecordsUrl loop).
  • Optional: registry table driving object list for the orchestrator pattern.

D · Measures and orphan logic

  • Per-object match rate: POCs with ≥1 related row ÷ POC count (define consistently).
  • Orphans: POC SID with no matches across the in-scope union (document the rule).
  • Refresh status measure or card bound to last refresh error table if you log errors.

E · Report shell

  • One Executive summary page: POC count, objects loaded, overall match rate, orphan count, refresh status.
  • Second page optional: per-object matrix or small multiples if time allows.
  • Apply brand tokens from the brand guidelines guide for colors and KPI layout.

F · Steward handoff

  • Table visual: POC name, SID, object columns or “no coverage” flag; enable export.
  • Document the orphan definition in the report tooltip or a one-line text box.

3. M query ranges (quick pointer)

Use queries-job-aide.html on the Task 2 tab. Prioritize bands called out in the Task 2 job aide: 201–240 (auth and REST), 241–280 (chunking and POC load — align to SID), 281–340 (child pulls), 341–400 (deltas and quality). Use the Advanced M · DAX · Settings tab for Desktop stability if refresh is flaky.

4. Decision checkpoints

SignalIf failingBefore adding scope
Refresh timeout Narrow SELECT lists, add pagination everywhere heavy, split staging tables Do not add new objects until one full refresh succeeds
Match rate > 100% or null Fix relationship cardinality and filter direction; confirm POC grain Freeze measures before new visuals
Orphan list empty but known gaps exist Verify SOQL filters use the same Id set as POC Master; check case and 15/18 char joins Re-run Gate 1 on data (playbook) before UI work

Phase 2 exit checklist

  • Full refresh completes within workspace limits (or documented manual refresh path).
  • Executive summary shows the five headline metrics with correct definitions.
  • At least one child object uses pagination or batching in production-like volume.
  • Exportable orphan table with POC_Record_Name and SID.
  • Short build note captured (what shipped, what is deferred to Phase C).

5. What comes next (Phase C)

After this slice is stable, move to Task 1 (playbook Phase C): RLS smoke tests, one permission or drift visual, and optional mobile layout. Keep Task 2 refresh on a schedule so regressions show up early.
