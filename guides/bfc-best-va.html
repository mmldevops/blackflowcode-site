0Executive framing

  • Single grain rule: every fact table row answers one question at one granularity; spreadsheets are normalized in Power Query before they touch the model.
  • Id integrity: all Salesforce keys pass the same trim / case / length / dedupe pipeline so joins never “silently fail.”
  • Provable refresh: gateway credentials, bounded API calls, and logged errors mean the dashboard is an operational asset, not a demo-only artifact.
  • Defensible metrics: DAX measures carry one definition, documented in tooltips; validation tables prove row counts and orphan logic.

1Spreadsheet governance

Structure

  • Use an Excel Table (Insert → Table) with headers on row 1 only; no merged cells in the header region.
  • One logical table per worksheet for anything loaded to Power BI; avoid “dashboard” layouts with blank spacer rows.
  • Keep SID (or raw Id column) and POC_Record_Name adjacent; avoid blank columns between keys and labels.
  • Do not embed subtotals or outline groups in source sheets used for refresh; PQ will treat them as bad rows unless filtered.

Data types in Excel

  • Format Ids as Text before paste; Excel’s Scientific notation corrupts Ids.
  • Dates as real date serials, not text like “01-Feb-25” mixed with “2025-02-01”.
  • Currency and counts as numbers; never mix “N/A” in a numeric column—use empty cell or a dedicated Status column.

2Salesforce Ids — full normalization playbook

Why 15 and 18 characters both appear

Salesforce exposes a 15-character case-sensitive Id in the UI and URLs, and an 18-character case-safe Id (checksum suffix) in APIs and exports. Mixing them without normalization produces missed matches when joins assume exact string equality.

Step-by-step (repeat for every Id column)

  1. Trim leading and trailing spaces (Text.Trim).
  2. Remove hidden characters (CHAR 160 non-breaking space, line breaks from CSV): optional Text.Clean after trim.
  3. Uppercase for consistent comparison (Text.Upper).
  4. Validate length: keep rows where length is 15 or 18; route others to an Exceptions query for stewardship.
  5. Deduplicate on the canonical key column before any merge to child data.
  6. Choose one join key for the model: either normalize all to 18 (preferred when mixing sources) or normalize all to first-15 with consistent casing—do not mix strategies across tables.

Power Query pattern (conceptual)

// After rename to SID
#"Trim Upper" = Table.TransformColumns(Source,{{"SID", each Text.Upper(Text.Trim(_)), type text}})
#"Valid Len" = Table.SelectRows(#"Trim Upper", each Text.Length([SID]) = 15 or Text.Length([SID]) = 18)
// Optional: add column Is18 = Text.Length([SID]) = 18
#"Distinct POC" = Table.Distinct(#"Valid Len", {"SID"})

For 15→18 conversion, implement a single tested M function (checksum algorithm) in one query and invoke it everywhere; do not duplicate logic across object queries.

Join-site discipline (REST / SOQL)

  • SOQL IN lists: batch (~200 Ids) to respect URL and governor practical limits.
  • When matching polymorphic lookups (WhatId), filter by Type prefix or object-specific subqueries to avoid wrong joins.
  • Always log the batch index and HTTP status in a diagnostic table on failure (empty table schema still in model).

3Power Query — transformation order & quality bar

Recommended step order

  1. Source → preserve raw query as duplicate for forensics.
  2. Promote headers / choose row 1.
  3. Rename to stable canonical names.
  4. Type detection then explicit types.
  5. Clean keys (Ids, dates).
  6. Filter to business scope.
  7. Merge / append.
  8. Remove columns not needed for visuals (reduces VertiPaq).
  9. Disable load on helper queries used only as functions.

Anti-patterns to avoid

  • Hard-coding org URLs or tokens in M text.
  • Merge before typing (hides type errors).
  • Loading wide “select star” REST extracts for every object.
  • Using Table.ReplaceValue on whole table for one-off fixes without documenting rule.

Query folding: for SQL and OData sources, keep filters early; defer irreversible steps that break folding until after filtered rows. For Salesforce REST via Web.Contents, folding does not apply—optimize with selective SOQL columns and server-side filters instead.

Buffered lists: when the same POC Id list feeds many invocations, materialize once and use Table.Buffer on the Id table where profiling shows repeated evaluation cost.

4Validation rules (PQ + model)

In Power Query

  • Build a dedicated Validation / Exceptions query: invalid Ids, blank keys, duplicate POC rows, SOQL rows with missing required fields.
  • Use Column quality and Column distribution in Power BI to confirm expected distinct counts before closing the editor.
  • Assert referential intent: left anti join POC vs child aggregate to produce orphan candidates (should match DAX definition later).

In the model (DAX validation tables)

  • Hidden table _Metrics_Check with measures: total rows per fact, distinct POC count, distinct join keys in bridge tables.
  • Card visuals on a hidden “Diagnostics” page for authors only—surface refresh duration and error row count.
  • Cross-check: sum of child rows by object should never be negative; match-rate never above 100% unless definition explicitly allows “multi-match.”

Composite model caution: mixing Import and DirectQuery on related tables requires expert modeling to avoid ambiguous paths and excessive DQ queries. For the POC program, prefer a single well-partitioned Import unless latency mandates otherwise.

Relationship checklist

  • One active relationship path between any two dimensions and a fact; know why inactive relationships exist and which USERELATIONSHIP measures use them.
  • Avoid bi-directional filters unless necessary; they can introduce circular ambiguity and slow SUMMARIZECOLUMNS.
  • Cardinality: many-to-one from fact to dimension; never “fix” wrong totals by flipping to many-to-many without a bridge.

6Advanced FN patterns (orchestrated M)

Align implementation with Multi-object Salesforce FN guide and numbered patterns in queries-job-aide.html (Task 2 bands).

  • Single SOQL runner function (FN): one (query as text) as table entry point handling headers, token injection, and try ... otherwise returning metadata columns IsError, ErrorReason, BatchIndex.
  • Registry-driven loop: table of objects, anchor field, optional WHERE clause template—invoke FN per row with Table.AddColumn + Table.ExpandTableColumn or List.Generate for controlled concurrency.
  • Error isolation: one failing object must not null the entire evaluation; union good tables and append error rows to a diagnostic query.
  • Pagination: wrap nextRecordsUrl in the FN or a sibling function; cap max iterations with a parameter to avoid infinite loops.
  • Rate limits: backoff on HTTP 429; optional Function.InvokeAfter spacing between calls when throttled.

7DAX — measures, safety, performance

  • Prefer measures over calculated columns unless the column is a physical slice needed for relationships or sort-by.
  • Variables at the top of complex measures: improves readability and avoids multiple scans of the same table expression.
  • Iterator discipline: SUMX over large fact tables is expensive; aggregate first with SUMMARIZE or use relationship-based SUM where possible.
  • Divide safely: use DIVIDE for match rates to avoid divide-by-blank.
  • Context transition: know when CALCULATE adds filters; test with COUNTROWS on a debug measure in a table visual.
  • Time intelligence: requires a contiguous date table marked as date table; never use FACT[date] directly for TOTALYTD without verifying gaps.
POC Count :=
DISTINCTCOUNT ( POC_Master[SID] )

-- Counts POC keys that have at least one child row (adjust Child[AccountId] if different)
Matched POCs :=
SUMX (
    VALUES ( POC_Master[SID] ),
    VAR CurrentKey = POC_Master[SID]
    RETURN
        IF ( CALCULATE ( COUNTROWS ( Child ), Child[AccountId] = CurrentKey ) > 0, 1, 0 )
)

Match Rate :=
DIVIDE ( [Matched POCs], [POC Count] )

Adjust table names to your model. Align this logic with your Power Query orphan anti-join for zero surprises.

Row-level security (RLS)

  • Author roles on a security table (username → LOB / region) with static data or limited refresh; avoid expensive dynamic RLS that evaluates per row unless required.
  • After any role change, run View as with representative users and compare measure totals to an unfiltered baseline.
  • Document that match-rate and orphan measures are evaluated after RLS filters—executives see the slice they are allowed to see, not global org numbers, unless you explicitly use REMOVEFILTERS in a governed measure.

8Conditional formatting & visual QA

Matrix / table visuals

  • Use Rules on match-rate columns: 0–50% red, 50–85% amber, 85–100% green (tune to org expectations).
  • Apply data bars on numeric intensity columns (child counts) with axis max set to p95 to avoid outlier washout.
  • Turn on Alt text for key visuals for accessibility and audit screenshots.
  • Icon sets: limit to three states; redundant icons plus color harms clarity in executive views.

Management-ready polish

  • One KPI definition per card; subtitle shows period and filter scope.
  • Synchronize slicers across pages for demo flow.
  • Use default bookmark for “clean” opening state.

9“Updating columns” & native query boundaries

  • Import: add or rename columns in Power Query, then refresh; avoid renaming columns in the model after reports reference them.
  • DirectQuery: prefer changing a database view or adding a computed column in the warehouse; PQ “added column” may not fold—test in Performance Analyzer.
  • Native SQL / SOQL in connector: document every hand-authored query in source control; parameterize dates and Id lists; never concatenate unsanitized user text into SOQL (injection and broken queries).
  • Schema drift: when a source removes a column, PQ refresh fails loudly—good. Add defensive Table.SelectColumns with MissingField.Ignore only when business agrees to soft-drop.

10Optimizing Power BI (full pass)

Model size (VertiPaq)

  • Remove high-cardinality text not shown in reports.
  • Split datetime into date-only column when time-of-day unused.
  • Round currency to business precision in PQ if sub-cent noise is useless.
  • Use integers for keys where possible.

Refresh & gateway

  • Parallelize only where gateway CPU and source limits allow.
  • Stagger large queries; sequence token retrieval before fan-out.
  • Incremental refresh on large facts when licensed and history rules are clear.
  • Performance Analyzer (Desktop): capture DAX query ms vs visual display; fix top 3 slow visuals first.
  • Aggregations: consider for large DirectQuery facts against imported summary tables—only with clear matching on granularity.
  • RLS: test with View as roles; RLS can change join paths—retest match-rate measures per role.

Dataset settings (service) worth reviewing

  • Scheduled refresh window vs upstream Salesforce maintenance; failure notifications to owners and delegates.
  • Query timeout and parallel loading (when available)—tune after observing gateway CPU and source throttling.
  • Enhanced metadata format and Large dataset storage format when models exceed default size thresholds (tenant dependent).
  • Deployment pipelines test stage: run the same zero-defect checklist before production promotion.

Incremental refresh (when licensed)

  • Define RangeStart / RangeEnd parameters and filter facts in PQ; archive cold history only if business agrees.
  • First refresh after policy change can be long—communicate before management demos.
  • Detect deleted rows policy: “real” deletes in source may not flow unless modeled (e.g., tombstone column).

11Zero-defect gates & audit trail

Pre-publish gate (all must pass)

  • No unresolved errors in Power Query; Exceptions query is empty or explicitly signed off.
  • POC Master: distinct SID count matches stewardship expectation within documented tolerance.
  • Full refresh completes in SLA; diagnostic table shows no 401/403 unless expected in dev.
  • Match rate and orphan definitions match between PQ validation, DAX measures, and footnotes on the report page.
  • Sensitivity labels and workspace permissions aligned to data classification.
  • Version note captured: dataset name, publish date, and change summary for auditors.

After findings (post-release discipline)

  • Log every production defect with root cause class: Id hygiene, relationship, DAX context, SOQL batching, gateway credential, or training.
  • Feed recurring defects back into this checklist—treat the checklist as a living control.
