Enhancements playbook

How to turn the full enhancement catalog into a controlled build and a credible mid-review story. Use this page for sequencing, ownership, and outcomes; use enhancements.html for the detailed card list, POC schema, child-object matrix, downloads, and connection templates.

Companion to enhancements.html · Mid-review May 26, 2026
Two documents, one workflow. Treat enhancements.html as the backlog and evidence catalog. Treat this playbook as the operating guide: what to do first, who cares, and how you will prove value in the review without boiling the ocean.

1. Guiding principles

  • Task 2 first: prove POC Master to child-record coverage before polishing Task 1 executive chrome.
  • No new Salesforce fields: every enhancement must be achievable with reports, REST, and Power BI model design only.
  • Three demo-ready outcomes: match rate by object, orphan list, and one governance view for Task 1 (for example permission drift or RLS proof).
  • Stop on green: if refresh is stable and match KPIs render, defer nice-to-have visuals rather than risking the demo.

2. Phased plan (suggested)

Phase A — Freeze scope (half day)

  • Lock POC Master columns: SID, POC_Record_Name.
  • Confirm in-scope child objects from the catalog in enhancements (verify API names in org).
  • Pick three Task 2 enhancements marked Critical in the catalog (chunking, ID normalization, match scorecard).
  • Pick two Task 1 enhancements for the review (for example RLS + one matrix or drift view).

Phase B — Vertical slice (Task 2)

  • OAuth cached once; SOQL batched; pagination on at least one heavy object.
  • Executive summary page: POC count, objects loaded, overall match rate, orphan count, refresh status.
  • Exportable orphan table for stewards.
  • Detailed guide: bfc-doc-p2 — Phase 2 (workstreams, checkpoints, exit checklist).

Phase C — Task 1 slice

  • RLS roles smoke-tested with View as.
  • One high-signal visual: permission matrix or profile compare or drift.
  • Mobile or PDF path only if time remains.

Phase D — Review rehearsal

  • Five-minute narrative: problem, data boundaries, live refresh, risks mitigated.
  • Backup static screenshots if gateway or Salesforce is unstable.
  • Capture build checklist from the enhancements download panel.

4. Decision gates (before you add more scope)

  • Gate 1 — Data: Are all POC SIDs valid 15 or 18 characters and deduplicated? If not, stop feature work until fixed.
  • Gate 2 — API: Does a full refresh finish under the workspace timeout? If not, narrow columns, add incremental, or split staging.
  • Gate 3 — Narrative: Can you explain each visual in one sentence tied to a risk or KPI? If not, remove the visual from the demo path.

5. Evidence checklist (for mid-review)

  • Screenshot or recording of successful scheduled refresh (or gateway test).
  • Table or export of top ten orphan accounts with POC names.
  • One RLS test matrix: role name, test account, expected row count.
  • List of in-scope Salesforce objects with match rate column (even if some are pending).
